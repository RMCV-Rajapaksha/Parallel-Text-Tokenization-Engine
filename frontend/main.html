Algorithm Design & Complexity
BPE Algorithm (Sennrich et al., 2016)

Byte Pair Encoding tokenizes text by iteratively merging the most frequent adjacent symbol pair. Starting from individual characters, each merge step creates a new subword unit. After N merges, words are represented as sequences of learned subword tokens.

low → l o w </w>
lower → l o w e r </w>
After merging l+o:
lo w </w> , lo w e r </w>
Complexity Analysis
VariantPair CountMerge ApplyKey Notes
SequentialO(V·L)O(V·L)V = vocab size, L = avg token len
OpenMPO(V·L / T)O(V·L / T)T threads; merge-table reduction O(T·H)
MPIO(V·L / P)O(V·L)P ranks; apply replicated on all ranks
CUDAO(V·L / G)O(V·L)G GPU threads; PCIe transfer per step
Bugs Fixed in Hybrid Version
#FileBugFix
1 bpe_mpi_cuda.cu MPI_MAXLOC picks rank with highest local count → wrong merge MPI_Gatherv compact pair lists → rank 0 sums counts → broadcast true winner
2 bpe_mpi_cuda.cu Full word-array re-uploaded to GPU per step (~14 MB × 500 = 7 GB PCIe) apply_merge_kernel applies merge in-place; only ~68 bytes cross PCIe per step
3 bpe_cuda.cu, bpe_mpi_cuda.cu __syncthreads() inside divergent probe loop → UB / warp deadlock Removed; atomicCAS/atomicExch provide necessary ordering
4 bpe_mpi_cuda.cu Full 512K-slot pair tables sent over MPI (~36 MB/step/rank) Only non-empty slots packed into CPair list sent via MPI_Gatherv
Bottlenecks & Future Improvements
Current Bottlenecks
  • OpenMP: Hash-table merge phase is serial → limits scaling beyond ~8 threads
  • MPI: Full-vocab apply after broadcast replicated on all ranks
  • CUDA: PCIe round-trip per step dominates for small vocabularies
Proposed Improvements
  • Use a min-heap / priority queue for O(log N) best-pair lookup
  • Incremental update: only recount pairs involving the newly merged symbol
  • GPU: keep vocabulary fully on device; only copy merge rule strings
  • MPI + OpenMP hybrid: each MPI rank uses multiple OMP threads
